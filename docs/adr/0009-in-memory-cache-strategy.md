# ADR-0009: In-Memory Page Cache for GET Endpoints

Date: 2026-03-21

## Status

Accepted

## Context

Caching options for a read-heavy REST API:

- **No caching**: Every request hits the database. Simplest, but provides no latency benefit for repeated reads.
- **HTTP cache headers (ETag / Last-Modified)**: Client-side conditional requests. Correct implementation requires generating and validating ETags, handling `304 Not Modified` — non-trivial for a collection endpoint.
- **In-memory page cache (server-side)**: Transparent to clients. Responses are stored in-process. Zero infrastructure overhead.
- **Distributed cache (Redis / Memcached)**: Shared across instances, survives restarts. Requires infrastructure and adds operational complexity.

This is a single-instance PoC with no horizontal scaling requirement. Infrastructure minimalism is an explicit constraint (see ADR-0004).

## Decision

We will use `gin-contrib/cache` with `InMemoryStore` and a 1-hour TTL for all GET endpoints. Write operations (POST, PUT, DELETE) explicitly invalidate affected cache entries before executing, via a `ClearCache` middleware wrapper, ensuring the next GET always returns fresh data.

Cache keys are derived from the full request URL using `cache.CreateKey()`, which matches the keys generated by `cache.CachePage()` exactly — ensuring invalidation targets the right entries.

## Consequences

**Positive:**

- Zero infrastructure: no Redis, no external process, no additional Docker service.
- Significant latency reduction for repeated GETs within the TTL window.
- Explicit, pre-emptive cache invalidation on writes — the cache is cleared before the handler runs, not after.
- Key derivation is centralized in `ClearCache`, consistent with `CachePage`.

**Negative:**

- Cache is not shared across multiple instances; horizontal scaling would require migrating to a distributed cache.
- Cache is lost on process restart — cold start serves all requests from the database.
- `GET /players/:id` cache entries are **not** invalidated by PUT or DELETE (only the collection and squad-number-keyed entries are busted). A UUID-keyed response can be stale until TTL expiry after a mutation.
- TTL-based expiry only — no event-driven invalidation for changes made directly to the database outside the API.
